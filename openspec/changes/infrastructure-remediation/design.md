## Context

The project implements an on-demand WireGuard VPN using Azure Container Instances, orchestrated by Azure Functions. The infrastructure scaffolding was generated by a prior AI toolchain and contains critical bugs that prevent any deployment from working:

- Credentials are populated with resource IDs/names instead of actual secrets or managed identity patterns
- The VPN container is deployed as standing infrastructure in the main Bicep stack (contradicting the on-demand model)
- The Function App cannot start due to an invalid storage connection string
- The container health probe targets a non-existent HTTP server
- No application code (Azure Functions) exists
- The container image cannot be built because the scripts it COPYs don't exist
- ACR is wired up with fake credentials and adds cost/complexity for a single public image

This design documents how each of these is fixed and the rationale behind each decision.

## Goals / Non-Goals

**Goals:**
- `az deployment group create` produces a working, correctly-wired Azure environment
- VPN container removed from standing infrastructure; `vpn-container.bicep` retained as a reference template
- Four Azure Functions implemented in JavaScript/Node.js (StartVPN, StopVPN, CheckVPNStatus, AutoShutdown)
- WireGuard container image is buildable, hosted publicly on GHCR, and passes health checks
- Monthly infra cost reduced by removing ACR, Premium Function plan, and dev private endpoints
- All OpenVPN surface area removed

**Non-Goals:**
- Web UI or client-facing portal
- Multi-region or HA deployment
- Advanced WireGuard features (split tunneling, custom DNS)
- Migrating existing deployed resources (none exist — this is first deployment)

## Decisions

### Decision 1: Remove ACR — use public GHCR image

Remove the `acr` module from `main.bicep`. The WireGuard container image contains no secrets — it is Alpine Linux + `wireguard-tools` + shell scripts. Keys, configs, and credentials are all injected at runtime from Key Vault and Storage.

Host the image publicly on GitHub Container Registry (`ghcr.io/<org>/az-demand-vpn-wg`). ACI references the image with no `imageRegistryCredentials` block. Pull is anonymous.

**Why over ACR + managed identity**: ACR adds ~$5/month, a build/push pipeline, an `AcrPull` role assignment, and a user-assigned managed identity wiring to every ACI deployment. For a single, rarely-changing image with nothing sensitive in it, that overhead is not justified.

**TODO (future)**: Add ACR as an optional private registry path when the team needs vulnerability scanning, geo-replicated pulls, or a private image. At that point, swap the image reference to `<acr>.azurecr.io/...` and add the managed identity pull role assignment to `vpn-container.bicep`.

---

### Decision 2: Azure SDK for ACI lifecycle — not ARM deployment API

`StartVPN` uses `@azure/arm-containerinstance` (`ContainerGroupsOperations.beginCreateOrUpdateAndWait()`) to create the ACI container group at runtime. `StopVPN` calls `containerGroups.beginDeleteAndWait()`.

**Why not ARM deployment API (Bicep at runtime)**:
- Requires `Microsoft.Resources/deployments/write` on the RG — close to Contributor, far broader than needed
- ARM deployment history persists parameter values; sensitive values (storage keys) appear in deployment records visible to anyone with `deployments/read`
- Concurrent `StartVPN` calls with the same deployment name conflict
- `@azure/arm-containerinstance` only requires `Microsoft.ContainerInstance/containerGroups/write` — least privilege

`vpn-container.bicep` is retained in `infra/modules/` as a readable reference spec for what the Function creates, but is not executed at runtime.

---

### Decision 3: Managed identity for Functions → Storage and ACI → Storage

**Function App → Storage**: Use `AzureWebJobsStorage__accountName` + `AzureWebJobsStorage__credential: managedidentity`. Grant the Function App's system-assigned managed identity:
- `Storage Blob Data Owner`
- `Storage Queue Data Contributor`
- `Storage Table Data Contributor`

**ACI → Storage (Azure Files volume mount)**: ACI currently requires a storage account key for Azure Files volume mounts — managed identity file-share mounts are not supported by ACI. Use `listKeys()` in Bicep at deploy time to retrieve the key. The key is used only to configure the volume mount resource property; it is not stored in app settings or Key Vault.

---

### Decision 4: Flex Consumption plan for Function App

Replace Premium P1V2 (~$140/month) with Flex Consumption plan. Flex Consumption supports VNet integration — the only reason Premium was chosen.

**Trade-off**: Cold start of ~1-3s on first invocation after idle. Acceptable for a VPN start/stop flow where the user's tolerance is on the order of seconds anyway.

---

### Decision 5: WireGuard health probe via exec

Replace `httpGet` probe on port 8080 with:
```
exec: ['wg', 'show']
```
`wg show` exits 0 when the WireGuard interface is up. Port 8080 does not exist in the container — the existing probe causes permanent liveness failure and restart loops.

---

### Decision 6: Remove private endpoints from dev environment

Dev deployments use public access with network ACL restrictions instead of private endpoints. Private endpoints add ~$7–14/month per endpoint in dev and require a VNet-connected machine to reach Storage and Key Vault during local development. Private endpoints are retained in prod parameters.

---

### Decision 7: Remove all OpenVPN surface area

The project is WireGuard-only. Remove UDP 1194 from NSG rules and the `openvpnPort` parameter from all modules. Dead config creates confusion and unnecessary attack surface.

---

### Decision 8: Function App runtime Node.js 20 LTS

Set `FUNCTIONS_WORKER_RUNTIME=node`, `WEBSITE_NODE_DEFAULT_VERSION=~20`. Node.js 20 is the current LTS and is supported by Azure Functions v4. The Bicep default was incorrectly set to `dotnet`.

## Risks / Trade-offs

- **ACI file-share key in ARM state** → The storage key from `listKeys()` is embedded in the ACI container group resource definition in ARM state. Restrict `Microsoft.ContainerInstance/containerGroups/read` to the operations team. Revisit if ACI adds managed identity file-share support.

- **Public container image reveals internal scripts** → Shell scripts in the image are readable by anyone. Mitigation: scripts contain no secrets; treat them as open source. If this becomes a concern, switch to private GHCR (free for org members) without needing ACR.

- **Flex Consumption cold start** → First VPN start after idle is 2-4s slower than Premium. Document in user-facing messaging ("VPN is starting…").

- **`vpn-container.bicep` not deployed via Bicep** → Bicep linter will warn about an unreferenced module if included in the project. Add `// runtime-only: used by StartVPN function via @azure/arm-containerinstance` comment and suppress the linter rule.

- **GHCR public image pull rate limits** → GHCR has no rate limits for public image pulls. Docker Hub does (100 pulls/6h per unauthenticated IP) — do not use Docker Hub.

## Migration Plan

This is a greenfield deployment — no existing Azure resources to migrate. Steps:

1. Fix Bicep bugs (P0-2 through P0-8)
2. Remove `vpnContainer` module from `main.bicep`; remove ACR module
3. Build and push WireGuard image to GHCR
4. Create container scripts and verify `docker build` succeeds locally
5. Deploy infra with `az deployment group create --what-if` first
6. Implement and test Azure Functions locally with Azurite
7. Deploy Functions; run end-to-end smoke test (StartVPN → CheckVPNStatus → StopVPN)
